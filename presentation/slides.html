Why normal autoscaling is a problem

  • When traffic spikes, systems often scale up (add servers)
  • They don't distinguish between real users and attackers
3 / 20

What goes wrong during an attack

  • During a DDoS or brute force, traffic looks like high load
  • Autoscaler adds more servers → you pay for attack traffic
  • Real users can still suffer; cost goes up with no benefit
4 / 20

What we want instead

  • Normal traffic + high load → scale up (add servers)
  • Attack traffic → do not scale; block it instead

"Attack-aware" = we are aware it's attack and change behaviour

5 / 20

The idea in one sentence

We use ML to classify traffic as normal or attack; we only scale when it's normal, and we block attack traffic so we don't pay to scale on it.

6 / 20

Why microservices?

  • Modern systems are built as many small services (e.g. one for products, one for payments)
  • Traffic hits an entry point (gateway) before reaching those services
  • We put attack-aware logic at the gateway so the rest doesn't scale on attack
7 / 20

What is a microservice?

  • A small, focused part of a bigger system that does one job
  • Services talk to each other over the network (e.g. HTTP)
  • Example: Catalog = one service (products); Gateway = another (entry + defence)
8 / 20

How we do it (high level)

  • 1. Collect or observe traffic metrics (e.g. requests per second, errors)
  • 2. Use ML to classify: normal vs attack (and attack type)
  • 3. If attack → gateway blocks (429); if normal → allow

10 / 20

The ML part (without the maths)

  • Train a classifier on labelled data (normal vs attack)
  • Input: e.g. requests per second, error count
  • Output: "normal" or "attack" (and type: DDoS, brute force, etc.)
  • Gateway (or detector) uses this to decide: allow or block

The two microservices in this project

  • Gateway: Entry point. Forwards to Catalog when normal; returns 429 (block) when attack
  • Catalog: Serves product data. Doesn't decide normal vs attack

Minimal microservices system where we demonstrate attack-aware behaviour

13 / 20

What runs automatically

  • A detector periodically reads live metrics from the Gateway
  • Runs the ML model; if attack-like → tells Gateway to block
  • When traffic looks normal again → tells Gateway to allow

System can switch to defence and back without someone clicking buttons
